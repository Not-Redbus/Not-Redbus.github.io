01. Problem Statement

Frontend clients are currently using PUT for partial updates, which is inadvertently wiping out unsubmitted fields (e.g., timestamps, IDs). This causes data loss.

02. Business Rules (Gherkin)

Rule A: Destructive Replacement (PUT)

GIVEN a record exists
WHEN a user sends a PUT request
THEN replace the ENTIRE object with the payload
AND delete any fields not present in payload.

Rule B: Safe Merge (PATCH)

GIVEN a record exists
WHEN a user sends a PATCH request
THEN update only the specified fields
AND preserve all other data.
03. Constraints
  • Idempotency: POST requests must return 409 Conflict if the resource already exists.
  • Persistence: Session data is ephemeral.
